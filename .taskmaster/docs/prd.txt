# Overview
A simple, classic single-player Snake game implemented in Godot 4 as a 2D project. The player controls a snake on a top-down grid playfield, eating apples to grow longer while avoiding collisions with the walls and the snake's own body. The game is intentionally minimal in scope — a faithful, polished take on the classic arcade formula — and serves as a small, complete game suitable as a portfolio piece, a Godot learning project, and a baseline for future experimentation with the engine.

The product is for:
- Casual players who want a quick, familiar arcade experience.
- The developer, as a vehicle for learning Godot 4's 2D workflow, input handling, scene management, and persistence.

It is valuable because it delivers a complete, shippable game with a tight feature set, a clear win/lose loop, and persistent progression (high score) — all achievable in a small number of focused development passes.

# Core Features
1. **Grid-based playfield**
   - What: A bounded 2D top-down grid (e.g., 20x20 cells) that defines all snake and food positions.
   - Why: Snake's gameplay only makes sense on a discrete grid; it determines movement, collision, and rendering.
   - How: A fixed-size grid rendered as the background. All gameplay entities snap to cell coordinates.

2. **Snake entity with grow-on-eat behavior**
   - What: A snake composed of a head segment and an arbitrary number of body segments. The snake moves one cell per game tick in its current heading. When it eats an apple, one segment is added to its tail on the next move.
   - Why: This is the core mechanic of the genre — growth makes the game progressively harder via self-collision risk.
   - How: A list/queue of grid cells representing the snake's body. Each tick: insert new head cell in heading direction; if no apple eaten, pop tail; otherwise keep tail and spawn a new apple.

3. **Apple spawning**
   - What: Exactly one apple exists on the playfield at any time. When eaten, a new apple spawns in a random empty cell.
   - Why: The apple is the only growth driver and scoring driver.
   - How: Random selection from the set of grid cells not currently occupied by the snake.

4. **Collision detection (walls and self)**
   - What: The game ends when the snake's head enters a wall cell (outside the playfield boundary) or a cell occupied by its own body.
   - Why: This is the lose condition; defines the difficulty curve.
   - How: After each move, check the new head cell against playfield bounds and the body segment list.

5. **Absolute-direction input**
   - What: The player changes the snake's heading using arrow keys or WASD (Up / Down / Left / Right). Inputs that would reverse the snake directly into its own neck are ignored.
   - Why: The most familiar and accessible control scheme for Snake.
   - How: Godot Input actions mapped to direction vectors; queued and applied at the next game tick to prevent same-tick double inputs and illegal 180° reversals.

6. **Constant tick rate**
   - What: The snake moves at a fixed, constant interval throughout the run. There is no speed-up as the snake grows.
   - Why: Keeps difficulty progression purely about navigating the lengthening tail, not reaction speed.
   - How: A single Timer (or accumulated delta) driving the game tick; tick interval is a constant defined in code/config.

7. **Live score + persistent high score**
   - What: The current run's score is displayed during play. The all-time best score is persisted to local disk and shown on the main menu and game-over screen. Each apple eaten increments the score by a fixed amount (e.g., 1).
   - Why: Provides a sense of progression and a reason to replay.
   - How: Score lives in the game state; on game over, compare against the persisted high score and overwrite if exceeded. Use Godot's `user://` filesystem (e.g., a JSON or ConfigFile save) for persistence.

8. **Main menu**
   - What: The first screen shown on launch. Displays the game title, the current high score, a "Start" action, and a "Quit" action.
   - Why: Gives the game a finished, complete-feeling shape and a clear entry point.
   - How: A dedicated Godot scene that transitions to the gameplay scene on Start and exits the application on Quit.

9. **Pause**
   - What: During gameplay, pressing Esc or P toggles a pause overlay that freezes simulation and shows "Paused" plus a hint to resume.
   - Why: Standard player expectation; prevents accidental losses from interruptions.
   - How: Set the gameplay node's `process_mode` / pause state via Godot's pause system; overlay is a CanvasLayer.

10. **Game-over screen with restart**
    - What: When the snake dies, gameplay halts and a game-over screen appears showing the final score, the (possibly newly updated) high score, a Restart option, and a Return-to-Menu option.
    - Why: Completes the play loop and encourages replay.
    - How: A separate scene or overlay shown on game over; Restart re-instantiates the gameplay scene; Return-to-Menu transitions back to the main menu.

# User Experience
**User personas**
- *Casual arcade player*: Wants a quick game they can pick up in seconds. Cares about a clean visual presentation and chasing their high score.
- *Developer / learner (the project owner)*: Wants a small, complete project demonstrating idiomatic Godot 4 2D structure (scenes, input map, timers, persistence, UI).

**Key user flows**
1. *First-time launch*
   - App opens → Main Menu shown (Title, "High Score: 0", Start, Quit).
   - Player presses Start → Gameplay scene loads, snake appears in starting position, first apple spawns, snake begins moving.
2. *Standard play*
   - Player steers with arrow keys / WASD, eats apples, score increments on each apple.
   - Player presses Esc/P → game pauses; press again to resume.
3. *Death and restart*
   - Snake hits wall or self → simulation stops → Game Over screen shows final score and high score (highlighted if new best) → player chooses Restart (loops back into gameplay) or Return to Menu.
4. *Quitting*
   - From Main Menu, player chooses Quit → app closes. High score remains saved for next launch.

**UI/UX considerations**
- Grid cells, snake segments, and apple should be visually distinct and readable at a glance: solid colored rectangles or simple sprites are sufficient.
- HUD during play is minimal: current score in a corner, nothing else competing for attention.
- Menu, pause, and game-over screens should use the same simple visual language as gameplay — no heavyweight UI framework or art needed.
- Controls should feel responsive: input applied on the very next tick, never dropped, and never allowed to cause an instant 180° self-collision.
- Keyboard-only operation throughout; no mouse required.

# Technical Architecture
**Engine and platform**
- Godot 4 (latest stable). Project is configured as 2D.
- Target: desktop (Windows primary, but no platform-specific code beyond what Godot provides).
- No networking, no external services, no third-party plugins required.

**System components (Godot scenes / scripts)**
- `Main.tscn` — root scene / scene switcher. Owns transitions between Menu, Game, and GameOver.
- `MainMenu.tscn` — title, high-score display, Start and Quit buttons.
- `Game.tscn` — the gameplay scene. Contains:
  - `Playfield` node — owns the grid dimensions and renders the background / wall boundary.
  - `Snake` node — owns the body-segment list, heading, and movement logic.
  - `Apple` node — owns the current apple position and respawn logic.
  - `GameTick` (Timer) — emits ticks at a constant interval.
  - `HUD` (CanvasLayer) — displays the current score.
  - `PauseOverlay` (CanvasLayer) — shown when paused.
- `GameOver.tscn` (or overlay) — shown on death; displays final score, high score, Restart, Menu buttons.
- `HighScore.gd` (autoload / singleton) — loads, exposes, and saves the persistent high score.

**Data models**
- *Grid cell*: integer (x, y) coordinate within `[0, width) x [0, height)`.
- *Snake state*:
  - `body: Array[Vector2i]` — ordered list of cells from head to tail.
  - `heading: Vector2i` — current direction vector (one of UP, DOWN, LEFT, RIGHT).
  - `queued_heading: Vector2i` — next direction to apply on tick, validated to disallow direct reversal.
  - `pending_growth: int` — number of additional segments to grow on upcoming ticks (incremented when apple eaten).
- *Apple state*:
  - `position: Vector2i` — current apple cell.
- *Game state*:
  - `score: int` — current run score.
  - `is_paused: bool`.
  - `is_game_over: bool`.
- *Persistent state* (on disk):
  - `high_score: int` — single integer stored at `user://highscore.save` (ConfigFile or simple JSON).

**APIs and integrations**
- None external. Persistence uses Godot's `user://` virtual filesystem only.

**Infrastructure requirements**
- None beyond the Godot editor and exported binary. No servers, no accounts, no build infrastructure beyond Godot's built-in export.

# Development Roadmap
**MVP requirements**
The MVP is a complete, playable game with all of the features below. No feature listed here may be deferred — the MVP is intentionally small.

1. Project skeleton
   - Godot 4 2D project configured.
   - Input map: `move_up`, `move_down`, `move_left`, `move_right` (bound to arrows + WASD); `pause` (bound to Esc + P); `ui_accept`/`ui_cancel` for menu navigation.
   - Scene-switching root (`Main.tscn`).

2. Core gameplay scene
   - Grid playfield with visible bounds.
   - Snake that spawns at a fixed starting position, length 3, heading right.
   - Constant-rate tick that advances the snake one cell per tick.
   - Apple spawning, with respawn into an empty cell on eat.
   - Growth on eat (one segment per apple).
   - Collision detection against walls and self → triggers game over.
   - Input handling with same-tick queueing and 180°-reversal rejection.

3. Scoring + persistence
   - In-run score incremented per apple.
   - HUD showing current score during play.
   - Persistent high score loaded on launch and saved on game over if exceeded.

4. UI flow
   - Main Menu (title, high score, Start, Quit).
   - Pause overlay (toggled by Esc / P).
   - Game Over screen (final score, high score with "New Best!" indication when applicable, Restart, Return to Menu).

5. Visual presentation
   - Simple themed visuals: solid colored rectangles or basic sprites for snake body, snake head (visually distinct from body), apple, and playfield background. No animations required.

6. Export
   - At least one working desktop export (Windows) verified end-to-end.

**Future enhancements (explicitly out of scope for MVP)**

- Multiple lives / respawn.
- Wrap-around walls as a toggle.
- Turn-relative control scheme as a toggle.
- Maze / obstacle layouts and multiple levels.
- Local two-player mode.
- AI opponent snake(s).
- Power-ups (e.g., shrink apple, score multiplier, slow-mo).
- Sound effects and music.

# Logical Dependency Chain
The order below is chosen so that something playable exists as early as possible, with each subsequent step layering on top of a working game.

1. **Project + input setup** — create the Godot 4 2D project, configure the input map, set up `Main.tscn` as a thin scene router. (Foundation; nothing visible yet.)
2. **Grid and static rendering** — render the playfield bounds and a static snake + static apple at fixed cells. Confirms coordinate system, rendering scale, and camera framing. (First visible output.)
3. **Tick-driven snake movement (no growth, no collisions)** — add the Timer, move the snake one cell per tick in a fixed direction, render it moving. (First moving thing.)
4. **Input → heading changes** — wire arrow/WASD input to change heading, with queued application on tick and rejection of direct reversals. (First *playable* state — the player can steer.)
5. **Apple eating + growth** — detect head-on-apple, grow the snake, respawn the apple in an empty cell. (Core loop now functional but cannot end.)
6. **Wall and self-collision → game over state** — implement collision checks and a minimal "game over, press R to restart" placeholder. (Game now has a full loop end-to-end; everything after this is layering polish and UI.)
7. **Score HUD** — add the score counter, increment on apple eat, display via CanvasLayer.
8. **Persistent high score** — implement the `HighScore` autoload, load on launch, save on game over if exceeded.
9. **Main Menu** — build the menu scene with Start / Quit and high-score display; route through `Main.tscn`.
10. **Game Over screen** — replace the placeholder restart with a proper Game Over scene/overlay showing score, high score (with new-best indicator), Restart, and Return-to-Menu.
11. **Pause overlay** — add the pause toggle and overlay using Godot's pause system.
12. **Visual polish pass** — ensure the head is visually distinct from the body, colors are readable, HUD is positioned cleanly, menus share the visual language.
13. **Export verification** — configure and run a Windows export, launch the binary, confirm high score persists across separate launches of the exported build.

Each step from (2) onward leaves the project in a runnable state. The MVP is complete after step (13).

# Risks and Mitigations
**Technical challenges**
- *Same-tick input causing illegal 180° reversal.* If two direction inputs arrive between ticks (e.g., Right then Down then Left), naive handling can let the snake fold onto itself.
  - *Mitigation:* Maintain a single `queued_heading` that is validated against the current heading (reject if directly opposite); apply only at tick boundary.
- *Apple spawning into an occupied cell as the snake grows large.* Pure random retry becomes inefficient when most cells are filled.
  - *Mitigation:* Build the list of empty cells (grid cells minus snake body) and pick uniformly from it. Simple, bounded cost given small grid sizes.
- *Pause behavior interacting with the tick Timer.* Godot's pause system must be configured so the gameplay nodes pause but the pause overlay still receives input.
  - *Mitigation:* Set `process_mode` to `PAUSABLE` on gameplay nodes and `WHEN_PAUSED` (or `ALWAYS`) on the overlay; verify explicitly when implementing.
- *High score file corruption / first-launch absence.* The save file may not exist on first run, or may be malformed if manually edited.
  - *Mitigation:* On load, treat missing or invalid file as `high_score = 0`; never crash on read errors. On save, write atomically (write then rename, or accept that occasional corruption resets the score — acceptable for a single-int save).

**MVP scoping**
- *Risk of scope creep into "future enhancements."* The genre invites endless variants (mazes, multiplayer, power-ups, AI).
  - *Mitigation:* The "Future enhancements" list above is explicitly out of scope for MVP. Each enhancement is a candidate for a follow-up iteration only after the MVP exports cleanly.

**Resource constraints**
- *Solo developer learning Godot 4.* Risk of getting stuck on engine-specific patterns (autoloads, scene transitions, the pause system, input map, `user://` filesystem).
  - *Mitigation:* The Logical Dependency Chain is structured so the project is runnable from step (2) onward. Each step touches one engine concept at a time, making it easy to isolate and learn.

# Appendix
**Research findings (source material)**
The PRD is derived from `.firecrawl/snake-gameplay-extracted.md`, which summarizes the Snake genre. Key points from that source that informed scoping decisions:
- The genre's core is a continuously-moving line/snake that grows on eating food and dies on collision with walls or itself.
- Many mechanical variations exist (turn-relative input, speed scaling, multiple lives, mazes, enemy snakes). The MVP deliberately picks the most classic option for each axis.
- Top-down perspective is canonical and is adopted here.
- Simplicity and low technical requirements are part of the genre's enduring appeal — reinforcing the choice to keep the MVP small.

**Design decisions locked for this MVP**
| Axis | Decision |
|------|----------|
| Players | Single-player only |
| Wall behavior | Hitting a wall ends the run (no wrap-around) |
| Controls | Absolute direction (Arrow keys + WASD) |
| Visuals | Simple themed (snake body + apple), no animations |
| Speed | Constant tick rate, no speed-up |
| Lives | One life per run |
| Scoring | Live score + persistent local high score |
| Screens | Main Menu, Pause overlay, Game Over screen with Restart |

**Technical specifications (initial defaults — tunable in code)**
- Grid size: 20 cells wide x 20 cells tall (final value to be tuned during step 2 of the dependency chain).
- Cell size: 32 px square (tunable for window framing).
- Tick interval: ~150 ms per move (tunable for feel).
- Starting snake length: 3 segments.
- Starting heading: Right.
- Score per apple: 1.
- High-score save location: `user://highscore.save`.
